🔑 How to create a Personal Access Token

  1. Go to github.com and sign in.
  2. Click your avatar (top right) → Settings.
  3. In the left sidebar, scroll to Developer settings → Personal access tokens → Fine-grained tokens.
  4. Click Generate new token.
  5. Give it any name (e.g. Chronicles Workpad). Set expiry to 1 year.
  6. Under Repository access, choose Only select repositories and pick your workpad repo.
  7. Under Permissions → Repository permissions, set Contents to Read and write.
  8. Click Generate token, copy it, and paste it above.
⚠ Your token gives write access to your repo. Do not share it. It is only held in memory during this browser session — it is never stored on disk or sent anywhere except directly to the GitHub API.

🔐 Security & Safety — what you should know

  • Token storage: Your Personal Access Token is typed directly into the password field and kept only in the browser tab's memory (a JavaScript variable). It is never written to localStorage, cookies, or any server. When you close or refresh the tab, the token is gone. You must re-enter it each session.
  • Public page & public repo: Your writing content (the HTML file) is stored in a public GitHub repo and hosted publicly. Anyone with your GitHub Pages URL can read the workpad. Do not store truly private information you would not want public. If you want privacy, change the repo to Private in GitHub Settings (this disables free GitHub Pages but keeps the file safe).
  • Old browser / stale commit risk: If you open the workpad in two different browsers and only one has the latest content, the older browser may overwrite newer work when it commits. The commit system fetches the current file SHA from GitHub before writing, which prevents GitHub-level conflicts — but it will replace the file with whatever that tab's state is. Best practice: always reload the live GitHub Pages URL before starting a session in a new browser to pull the latest baked state.
  • Token expiry: Fine-grained tokens expire (you set the date). When yours expires, you will get a 401 error in the sync log. Simply generate a new token and paste it in.
  • Minimal permissions: Your token should only have Contents: Read & Write on the single workpad repository — never organisation-wide or account-wide permissions.

Maintenance tip: Every few months, go to github.com → Settings → Developer settings → Personal access tokens, review active tokens, and revoke any you no longer need. Rotate your workpad token once a year when it renews.

GitHub Guide — Hosting Your Workpad

This guide walks you through hosting this file on GitHub Pages — completely free, no technical experience needed. Once set up, your workpad will live at a URL you can open from any browser.

1 Create a GitHub account (if you don't have one)

Go to github.com and sign up. Then come back here to the ☁ GitHub Sync tab to push your writing directly.

2 Create a new repository

  1. Once logged in, click the green "New" button (or the + icon, top-right).
  2. Name the repository (e.g. chronicles-workpad).
  3. Set it to Public (required for free GitHub Pages hosting).
  4. Tick "Add a README file".
  5. Click "Create repository".

3 Upload this file the first time

  1. Inside your new repository, click "Add file" → "Upload files".
  2. Upload this HTML file. Name it exactly: index.html
  3. Click "Commit changes".
💡 After the first upload, you can use the ☁ GitHub Sync tab in this workpad to commit directly without ever touching GitHub's website again.

4 Turn on GitHub Pages

  1. In your repository, click Settings.
  2. Scroll down the left sidebar to Pages.
  3. Under "Source", select Deploy from a branch.
  4. Choose branch: main, folder: / (root).
  5. Click Save. Your live URL will appear after a minute or two.

Your workpad will be live at:

https://[your-username].github.io/chronicles-workpad/

5 Saving your writing going forward

Use the ☁ Save to GitHub button at the top of this page — or go to the ☁ GitHub Sync tab. This bakes your writing directly into the HTML file and commits it to GitHub. No more manual uploading.

✓ Every commit creates a versioned snapshot. Go to your repository → Commits to see the full history and restore any earlier version.

6 Export draft text to GitHub

In the ☁ GitHub Sync tab, you will find two export buttons:

  • Export Drafts as .txt — pushes draft_export.txt containing only the Draft prose from every chapter, in order. Clean manuscript text.
  • Export Full (Draft + Ideas + Meta) as .txt — pushes full_export.txt containing Draft, Ideas & Concepts, and Meta-Data from all three tabs of every chapter. Useful for full backups or sharing with an editor who needs your notes too.
💡 These .txt exports are snapshots — they do not update automatically when you type. Push them whenever you want a fresh plain-text backup.

7 Viewing your commit history & comparing versions

To see all your commits (save history):

  1. Go to your repository on github.com.
  2. Click the clock icon (or the text showing "X commits") near the top-right of the file list — this opens the full commit log.
  3. Each entry shows the commit message, date, and time. Click any commit to see what changed.

To restore an earlier version of your workpad:

  1. In the commit log, click the commit you want to go back to.
  2. Click on index.html in that commit's file list.
  3. Click the "Raw" button to see the raw HTML.
  4. Save that page (Ctrl+S / Cmd+S) to download it, then open it locally — your writing from that point in time will be there.

To compare two commits (diff view):

  1. In the commit log, click any commit.
  2. GitHub shows a diff — red lines are what was removed, green lines are what was added. For this workpad, most of the diff will be the baked state JSON (your writing data encoded inside the file).
  3. For a human-readable comparison, use the Raw method above to download two versions and compare them side-by-side in any text editor.
💡 Write meaningful commit messages (e.g. "Chapter 4 — Tress reaches the coast") so your history is easy to navigate. The auto-generated messages include the date, time, and word count.

8 Safety when using multiple browsers

If you open the workpad on a different device or browser that has stale content (older than your last GitHub commit), committing from that browser will overwrite your latest work with the older version.

⚠ Always open the live GitHub Pages URL before starting a new session, rather than a saved local copy. The live URL always serves the latest committed version.

The workpad detects the current file SHA on GitHub before writing — this prevents two tabs from committing simultaneously to the same SHA — but it cannot protect you from intentionally pushing old content. When in doubt, reload first.

9 New features — what's been added (v4)

This version of the workpad includes the following additions on top of v3:

  • Chapter reordering — each chapter now has ↑ ↓ move buttons, plus "Insert Before" and "Insert After" buttons to add a new chapter in any position. The index updates automatically.
  • Tags field — each chapter has a new "Tags" meta field. Add multiple words or phrases as chips. Tags and Location/Setting are now shown in the Index tab alongside POV.
  • Timeline insert between events — timeline nodes now have an "Insert After" button to add a new event between any two existing events.
  • Full export (.txt) — new button exports Draft + Ideas & Concepts + Meta-Data from all chapters into a single full_export.txt on GitHub.
  • Dark Draft & Midnight Sepia themes — two new colour modes with fully dark writing areas, so text-on-dark is comfortable for long sessions. Choose via the six dots in the top bar.
  • Git commit log in Writing Session Log — every GitHub commit is automatically recorded in the Progress → Writing Session Log with the commit message, date, and time.
  • Security info box — the GitHub Sync tab now has a dedicated section explaining token storage, public page risks, multi-browser safety, and maintenance guidance.
  • Title updated — the workpad is now titled Chronicles of Ruloba.
